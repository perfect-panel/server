# V4.3 重置/断网生效 SLA(决策 26 修订)

## 现状

用户重置设备 / 管理员强制停用 / 流量超 24h 触发断网这三个动作,在 server 侧:
1. 写 DB(token+uuid 换新或 cut_off_at 设置)
2. `DEL server:user:list:*` 失活节点缓存

节点(`ppanel-node`)在自身 `pull_interval`(默认 60s)到点时拉新 user list,
看到旧 token/uuid 已不存在,立刻拒绝旧客户端。

**收敛 SLA:最坏 60s,典型 30s(因为 pull 时间均匀分布)**

## 为什么不做"主动 evict"

V4.3 落地方案 §4.5 写了:
> POST /v1/server/evict { uid: old_device_id, ip: "*" }

这个语义需要 **server → node 反向调用**,但当前节点协议是节点单向 pull,
节点本身不暴露 HTTP 端点接收 server 推送。

要做到 1s 收敛,有 3 条路,都不轻:

| 方案 | 工程量 | 运维复杂度 | 备注 |
| --- | --- | --- | --- |
| A. 节点暴露 HTTP callback | 中 | 高:节点要开端口,server 要配回调 URL | 防火墙/NAT 不友好 |
| B. Redis Pub/Sub 共享 | 低 | 高:节点机要能直连 server Redis | 跨 region 部署不友好 |
| C. push response 加 dirty bit | 低 | 低 | 节点要每次 push 后立即重拉,放大 N 倍 GET |

C 方案折中:不引入额外通道,只在节点已有的 push 流上夹带信号。
但要改节点端代码,且收敛延迟取决于 push 频率(可能比 pull 还慢)。

## 决策

**接受 60s SLA,写入用户协议 + FAQ。**

文案(决策 4.5 异步通知 + 13 客服 FAQ):

> Q: 我重置设备后旧 URL 还能用多久?
> A: 节点最快 1 秒内同步,最慢 60 秒内旧 URL 完全失效。

实测大多数节点在 30 秒内能感知;若某节点超过 90 秒未同步,
可能是节点 pull 失败,管理员可:
1. 手工去节点机重启 ppanel-node 容器
2. 或调短 pull_interval(代价:server 负载略升)

## 何时升级到 1s SLA

如果未来 V4.x 出现以下情况之一,再实施方案 C:
- 客户投诉"重置后旧 URL 还能用 30 秒"成为高频反馈
- 出现因 30 秒延迟造成的滥用案例(短时间内分发 URL 给多人)
- 节点 pull_interval 因性能压力被加大到 ≥ 5 分钟

届时建议:
- ppanel-server: 重置/断网生效后,POST `/v1/server/push` 响应里带 `force_reload=true`
- ppanel-node: 收到 `force_reload` 立即重拉 user list(替代等待下一次 pull tick)

## 监控建议

为了上线后量化实际 SLA,在节点端加 metric:
- `node_user_list_age_seconds`(gauge):当前内存中 user list 距上次成功拉取的秒数
- `node_user_list_pull_total`(counter):pull 次数(成功/失败/304)

server 端加:
- `server_user_list_cache_invalidate_total`(counter):每次 DEL 操作 +1

通过 Prometheus 计算 P95/P99 实际收敛延迟。
