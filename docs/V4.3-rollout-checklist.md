## 阶段 1:灰度 10%

策略:仅 1 台节点 + 1 个新套餐启用 V4.3 device-billing。

- [ ] 创建灰度套餐:`unit_price_per_device > 0`,挂 1 台测试节点。
- [ ] 内部账号买 3 设备 ¥10/台 → 收 ¥30 → 3 个独立订阅 URL(对齐 DoD #1)。
- [ ] 第 10 天加购 1 设备 → 收 ¥10×(20/30)=¥6.67(对齐 DoD #2)。
- [ ] 重置设备 → 旧 URL 立即失效;1 小时内重置 4 次 → 第 4 次 429 cooldown(DoD #12/#13)。
- [ ] 流量灌至 100% → 限速到 1 Mbps 24h → 加购 50GB → 立即恢复(DoD #6/#7/#8)。
- [ ] **多协议同时连接速率不倍增**(决策 32 修复点)。压测脚本同时拉 vless + ss + hy2,聚合带宽 ≤ 套餐 SpeedLimit。

## 紧急回滚预案(决策 12)

```sql
-- A. 回退到旧时长计费(保留 schema,只关闭新模型)
UPDATE subscribe SET unit_price_per_device = 0;  -- 让 purchaseLogic 走 legacy 分支

-- B. 重启节点回退到 taguuid 限速桶(若 limiter 修复影响现有用户)
-- ServerUser.UserSubscribeId = 0 时会自动 fallback,无需代码改动。
-- 服务端临时:
UPDATE servers SET direct_list = NULL;  -- 关闭直连白名单下发(若客户端崩)
DEL server:user:list:*  -- 强制节点重拉

-- C. 完整 rollback schema(高危,只在必要时)
-- 跑 02126_device_billing.down.sql
```
